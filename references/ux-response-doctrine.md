# UX 响应 Doctrine

本 doctrine 治理 `instrument-of-state` 在用户可见输出层（UX / response layer）的一切规范：包括每一次回复的信息密度与结论优先级、首次使用引导与用户控制点、以及四类常见任务的输出骨架。凡是面向终端用户的文字都在本 doctrine 的管辖范围内，目标不是增加制度感，而是让用户更快看懂、敢继续、知道下一步。

---

## Section A — 响应规范

### 核心原则

1. 先给结论，再给过程。
2. 先说用户要做什么，再说系统做了什么。
3. 中途更新极短，收口时再完整。
4. 技术异常写日志，不把内部错误直接甩给用户。
5. 官署名称可以保留，但必须配人话别名。

### 结论优先顺序

每次用户可见回复都优先回答这四件事：

1. 现在在哪个阶段
2. 现在能不能继续执行
3. 下一步是谁做什么
4. 如果需要用户选择，用户现在有哪些可选动作

这四件事应优先出现在：

- 朝政进度摘要行
- 门下省裁决卡
- 最终行动

### 朝政进度的用户优先写法

无论是完整看板还是简报看板，开头都优先给出以下信息：

- `当前阶段`
- `执行权限`
- `下一动作`
- `用户可选动作（如需要）`

如果没有用户决策点，不必硬写"用户可选动作"。

### 官署名称的人话别名

用户可见内容中，建议使用以下写法：

- `吏部（分工/责任）`
- `户部（成本/范围）`
- `礼部（文档/发布）`
- `兵部（应急/故障）`
- `刑部（验证/验收）`
- `工部（代码执行）`

三省也可使用更易懂的解释：

- `中书省（起草方案）`
- `门下省（审校裁决）`
- `尚书省（调度执行）`

### 模式的人话解释

不要只写模式名，用户可见时应加一层解释：

- `标准（先审后做）`
- `严格（多一层成本与合规门）`
- `紧急（先止血，后补审）`

### 错误与降级文案

不要把以下字样直接暴露给普通用户：

- `hook error`
- `PreToolUse`
- `permissionDecision`
- `Agent guard`

如果守卫失败但主流程仍可继续，统一改写为：

`治理守卫已降级，主流程继续。技术细节已写入本地日志。`

如果权限被拒绝，直接说人话原因，例如：

- `工部（代码执行）尚未解锁，因为门下省还未批准本案。`
- `门下省（审校裁决）需要先收到中书省正式奏折。`

### 信息密度控制

#### 首次实质性回复

- 可以完整
- 需要让用户建立心智模型

#### 中途更新

- 只保留最少必要信息
- 不要机械重复完整看板
- 不要把同一原因写三遍

#### 最终收口

- 可以重新完整
- 必须让用户知道已经做完什么、还剩什么风险

### 用户视角检查

在发送最终文案前，至少自检这三问：

1. 用户第一眼能看出是否可继续吗？
2. 用户不读长文也知道下一步吗？
3. 用户不懂官署术语也能理解大意吗？

## Section B — 首次使用引导与用户控制点

本节用于降低新用户第一次使用 `instrument-of-state` 的理解成本，并给用户可预期的停靠点。

### 首次使用引导

当满足以下任一条件时，可视为"可能是第一次使用"：

- 用户直接输入一个大任务，没有补充任何流程控制词
- 用户明显在试探插件怎么工作
- 当前会话中尚未出现过清晰的治理流程回复

此时在首个实质性回复末尾，加一条不超过两行的提示。

示例：

`使用提示：你可以随时说"继续执行""仅出方案"或"跳过宣示"，我会在对应边界停住。`

### 用户控制点

以下控制点应作为用户可理解的自然语言选项，而不是内部制度术语：

- `继续执行`：按当前已批准路径继续推进
- `仅出方案`：停在中书省/门下省阶段，不进入工部落地
- `跳过宣示`：完成本地交付，但不走飞书/Lark 发布
- `先看风险`：优先返回风险、边界和待决项
- `先别改代码`：允许调研和起草，不允许进入工部

### 展示时机

#### 应展示控制点的场景

- 门下省返回附条件批准
- 尚书省准备发令前
- 宣示决策尚未锁定时
- 用户尚未明确"要不要直接做"

#### 可省略控制点的场景

- 紧急止血已必须立即执行
- 用户已经明确说"直接做"
- 本案只是一个纯问答或纯文档总结

### 控制点写法要求

1. 控制点必须写得像按钮文案，不要像制度说明。
2. 一次最多给 3 个，避免选择过载。
3. 如果当前没有决策点，不要为了完整性硬塞控制点。

## Section C — 任务类型响应模板

本节为四类常见任务提供用户可见输出骨架。所有模板都遵循"先结论，后细节"的规则。

### 1. 故障 / 应急类

适用场景：

- 线上故障
- CI/CD 中断
- 热修复
- 回滚

优先强调：

1. 现在是否已止血
2. 是否还在影响用户
3. 下一步是谁处理什么

建议摘要写法：

- `当前阶段：兵部（应急/故障）牵头止血`
- `执行权限：可先止血，后补审`
- `下一动作：先恢复服务，再补门下审校与刑部验收`

### 2. 前端 / 设计类

适用场景：

- 页面改版
- 组件重做
- 视觉升级
- 交互优化

优先强调：

1. 当前是否还在方案阶段
2. 视觉方向是否已锁定
3. 是否进入真实实现

建议摘要写法：

- `当前阶段：中书省（起草方案）正在明确页面方向与交互目标`
- `执行权限：暂未进入工部（代码执行）`
- `下一动作：先确认视觉方向，再进入实现与验证`

### 3. 发布 / 宣示类

适用场景：

- 发布说明
- 飞书/Lark 文档
- 对外同步
- PR 仪式与交接

优先强调：

1. 本次是否需要正式宣示
2. 是现在发布、仅出文档，还是暂缓
3. 谁来发、发给谁

建议摘要写法：

- `当前阶段：礼部（文档/发布）正在准备正式文稿`
- `执行权限：可先产出文档，再决定是否通知收件人`
- `下一动作：确认收件范围与权限后进入宣示`

### 4. 文档 / 研究类

适用场景：

- 方案分析
- 代码调研
- ADR
- 文档整理

优先强调：

1. 当前是否只出方案，不动代码
2. 是否需要门下裁决
3. 用户是否还需要批准后续执行

建议摘要写法：

- `当前阶段：中书省（起草方案）正在整理结论与边界`
- `执行权限：当前仅产出方案，不直接落地代码`
- `下一动作：门下省审校后，再决定是否转入执行`

### 模板使用规则

1. Shangshu 应尽早判断任务类型。
2. 选定任务类型后，优先采用对应摘要骨架。
3. 即使采用模板，也不能跳过朝政进度、门下裁决卡和最终行动。

## Relation to other doctrines

UX 响应 doctrine 处于治理链条的最外层——它约束所有其他 doctrine 的结果如何呈现给用户。以下是与相邻 doctrine 的具体衔接点：

- **`cadence-doctrine.md`（节奏 doctrine）**：
  - 本 doctrine 的「结论优先顺序」四问（当前阶段 / 执行权限 / 下一动作 / 用户可选动作）正是 cadence 四态（发牌 / 留白 / 跳过 / 插队）在 UX 层的落地模板。
  - cadence 的「发牌」形态要求每次节点转换都要向用户交付一张可见的牌；本 doctrine 规定这张牌的文字形态——结论在前、术语配人话、模式带解释。
  - cadence 的「留白」「跳过」两态要求极短中途更新；本 doctrine 的「信息密度控制 - 中途更新」段是其具体文案约束。
  - 交叉引用：cadence-doctrine §发牌的形态规范 ↔ 本 doctrine §朝政进度的用户优先写法 / §官署名称的人话别名。

- **`deal-card-doctrine.md`（发牌 doctrine）**：
  - deal-card 规定「发什么牌」（三判断：是否减少不确定性、是否提高下一步清晰度、是否不过度打断）；本 doctrine 规定「牌面长什么样」——即用户控制点文案、模式人话解释、降级文案改写。
  - deal-card 的「发牌的形态规范」与本 doctrine 的「用户控制点 - 写法要求」共同约束用户可选动作：deal-card 决定是否要给选项，本 doctrine 决定这些选项如何写成按钮式文案且一次不超过 3 个。
  - 交叉引用：deal-card-doctrine §三判断 ↔ 本 doctrine §首次使用引导 / §展示时机。

- **`meta-unit-doctrine.md`（元单位 doctrine）**：提供上层判准，说明本 doctrine 为何应独立存在（用户可见层与治理层应可分别替换）。

## Meta-Unit Self-Check

按 `meta-unit-doctrine.md` 的四判断自检本 doctrine：

1. **独立陈述检查**：本 doctrine 陈述"用户可见输出应优先结论、配人话别名、给可选动作、按任务类型收束骨架"——这是一条可被独立表述的规则集，不依赖具体 agent/skill 实现即可理解。通过。
2. **定位检查**：治理链条中明确定位在最外层——UX / response 层。它在 cadence 决定"何时发牌"之后、deal-card 决定"发什么牌"之后，约束"牌的文字形态"。通过。
3. **替换代价检查**：若替换本 doctrine，影响面覆盖所有用户可见回复（进度摘要、裁决卡、最终行动、首次提示、任务模板）；但替换不牵动 cadence / deal-card / meta-unit 的结构。耦合是向下单向的，替换代价可控。通过。
4. **复用检查**：Section A 的结论优先顺序与降级文案改写可被 menxia-review、shangshu-dispatch、以及任何对外发言的 skill 复用；Section B 的控制点语义可被首次引导和附条件批准两类场景共享；Section C 的四类骨架可被 coordinator 路由直接套用。复用面覆盖 ≥3 个 agent/skill。通过。

四条全部通过，本 doctrine 作为独立元单位合格。
