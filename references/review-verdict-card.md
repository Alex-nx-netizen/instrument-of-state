# 审校裁决摘要卡片

当 review 返回审校结果时，先给用户一张可快速扫读的裁决卡片，再展开完整审查意见。

目标：

- 让用户先知道能不能执行
- 让用户先知道卡在哪里
- 让用户先知道下一步是谁做什么
- 避免一上来就是长篇审查正文

## 使用时机

以下场景优先显示裁决摘要卡片：

1. review 完成审校后的第一条用户可见回复
2. dispatch 引用 review 结论并准备发令时
3. review 返回附条件批准、退回、驳回时

## 卡片格式

使用简洁分段，不用大段散文。建议保留以下字段：

- `裁决`：批准 / 附条件批准 / 退回 / 驳回
- `当前阶段`：让用户知道现在卡在审校、补条件，还是已经转入执行
- `一句话结论`：用一句话说明当前是否可以推进
- `执行权限`：已解锁 deliver / 暂未解锁 deliver
- `下一动作`：明确谁在做什么
- `关键条件`：仅在附条件批准时列出 1 到 5 条
- `退回原因`：仅在退回或驳回时列出 1 到 3 条

## 批准模板

```md
## 审校裁决卡

- 裁决：批准
- 当前阶段：dispatch 准备发令
- 一句话结论：奏折结构完整、边界清楚，可以按现案发令。
- 执行权限：已解锁 deliver
- 下一动作：dispatch 按已审路线调度相关部门执行。
```

## 附条件批准模板

```md
## 审校裁决卡

- 裁决：附条件批准
- 当前阶段：dispatch 补条件中
- 一句话结论：方向可行，但必须先补齐下列条件后再执行。
- 执行权限：暂未解锁 deliver
- 下一动作：dispatch 先满足条件，再决定是否发令。
- 关键条件：
  1. <条件一>
  2. <条件二>
  3. <条件三>
```

## 退回模板

```md
## 审校裁决卡

- 裁决：退回
- 当前阶段：退回 draft 补稿
- 一句话结论：奏折尚不具备治理质量，需要退回 draft 补稿。
- 执行权限：暂未解锁 deliver
- 下一动作：draft 补齐缺失信息后重新报审。
- 退回原因：
  1. <原因一>
  2. <原因二>
```

## 驳回模板

```md
## 审校裁决卡

- 裁决：驳回
- 当前阶段：本路径停止推进
- 一句话结论：当前方案不应继续推进，应停止本路径。
- 执行权限：暂未解锁 deliver
- 下一动作：dispatch 停止执行，并向用户说明替代方向或终止原因。
- 退回原因：
  1. <原因一>
  2. <原因二>
```

## 写法要求

1. 先卡片，后正文。
2. 一句话结论必须是用户视角，不要写官样空话。
3. 下一动作必须可执行，避免"待定""后续再议"。
4. 条件数量控制在 5 条以内，按优先级排序。
5. 如果条件已满足，dispatch 应明确写"条件已满足，现转入执行"。
6. 如果用户不理解官署名称，应仅凭"当前阶段 / 执行权限 / 下一动作"也能看懂当前局面。
