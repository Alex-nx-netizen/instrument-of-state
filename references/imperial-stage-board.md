# 朝政进度

当 `dispatch` 技能准备用户可见的回复时，使用此文件。

朝政进度与团队蓝图互补，必须同时展示：

- 朝政进度：说明流程走到了哪一步（纵向时间线）
- 团队蓝图（参见 `team-blueprint-board.md`）：说明谁在做什么、用什么能力（横向组织图）

每个受治理的任务都应以可见的进度清单开始，让用户一目了然地看到：

- 哪些阶段已完成
- 哪些阶段正在执行
- 哪些阶段正在等待
- 哪些阶段被阻塞以及原因
- 哪些阶段已跳过

## 状态标记

使用以下中文状态词：

- `已完成`：该阶段已执行完毕
- `进行中`：该阶段正在执行
- `等待中`：该阶段尚未开始，正在等待前置条件
- `已阻塞`：该阶段因依赖未满足而无法推进
- `已跳过`：该阶段本次任务不需要

## 显示格式

使用待办清单（todolist）格式，用 markdown 复选框表示状态：

- 已完成的阶段：`- [x] 阶段名：说明`（打勾表示已完成）
- 已跳过的阶段：`- [x] 阶段名（已跳过）：原因`（打勾表示已处理）
- 进行中的阶段：`- [ ] 阶段名（进行中）：说明`
- 等待中的阶段：`- [ ] 阶段名（等待中）：说明`
- 已阻塞的阶段：`- [ ] 阶段名（已阻塞）：说明`

## 展示节奏

为了兼顾治理感和终端可读性，朝政进度分为两种展示层级：

1. 完整看板：
   - 用于首个实质性回复
   - 用于阶段发生明显跃迁时，例如起草完成、门下裁决返回、开始执行、准备收尾
   - 用于最终收口
   - 必须列出全部 14 个阶段
2. 简报看板：
   - 用于中间过程更新
   - 只保留 3 到 5 行关键信息：裁决状态、当前阶段、已完成的最近节点、阻塞点、下一步
   - 不重复展开全部 14 个阶段，避免终端刷屏

如果本次回复没有新的阶段变化，不要机械重复完整看板。

## 用户优先摘要

在完整看板或简报看板开头，优先回答以下问题：

1. `当前阶段`：现在卡在谁手里
2. `执行权限`：现在能不能继续执行
3. `下一动作`：接下来是谁做什么
4. `用户可选动作`：只有在存在真实决策点时才显示

如果用户不读后面的长文，也应该能仅靠这几行判断当前局面。

## 标题格式

在进度清单之前显示标题行：

> 国之重器 ｜ <请示摘要> ｜ 模式：<标准（先审后做）/严格（多一层成本与合规门）/紧急（先止血，后补审）>

## 阶段顺序

必须按以下固定顺序列出所有 14 个阶段：

1. 皇上下旨
2. 太子承旨
3. 锦衣卫侦察
4. 中书省起草
5. 门下省封驳
6. 尚书省发令
7. 吏部（分工/责任）
8. 户部（成本/范围）
9. 礼部（文档/发布）
10. 兵部（应急/故障）
11. 刑部（验证/验收）
12. 工部（代码执行）
13. 奏折回呈
14. 礼部宣示

## 规则

1. 必须列出完整的进度清单，即使某些阶段被跳过。
2. 必须单独列出六部中的每一部。
3. 清单必须反映当前实际进展，不能虚构理想流程。
4. 尚未启动的阶段标记为"等待中"并注明等待什么。
5. 不需要的阶段标记为"已跳过"并简要说明原因（打勾处理）。
6. 被阻塞的阶段标记为"已阻塞"并说明等待什么。
7. 正在执行的阶段说明谁在做什么。
8. 等待或阻塞的阶段指明等待的机构、裁决、卷宗或产物。
9. 首个实质性回复、里程碑跃迁、最终收口必须使用完整看板。
10. 中间过程更新优先使用简报看板，避免重复铺满终端。

## 进度清单模板

```
## 朝政进度

> 国之重器 ｜ <请示摘要> ｜ 模式：标准（先审后做）

- [ ] 执行权限：当前还不能进入工部（代码执行）
- [ ] 下一动作：中书省（起草方案）提交正式奏折后，转门下省审校
- [ ] 用户可选动作（如需要）：仅出方案 / 继续执行

- [x] 皇上下旨：用户已提交请示"<请示摘要>"
- [ ] 太子承旨（进行中）：`dispatch-agent` 正在判定模式、风险与是否立案
- [ ] 锦衣卫侦察（等待中）：等待太子承旨完成
- [ ] 中书省起草（等待中）：等待锦衣卫侦察结果
- [ ] 门下省封驳（等待中）：等待中书省返回奏折
- [ ] 尚书省发令（等待中）：等待门下省裁决
- [x] 吏部（分工/责任，已跳过）：本案无额外分工需求
- [ ] 户部（成本/范围，等待中）：等待尚书省决定是否需要成本评估
- [ ] 礼部（文档/发布，等待中）：等待尚书省决定是否需要正式文书
- [x] 兵部（应急/故障，已跳过）：本案非紧急事件
- [ ] 刑部（验证/验收，等待中）：等待尚书省决定是否需要合规校验
- [ ] 工部（代码执行，已阻塞）：等待门下省批准
- [ ] 奏折回呈（等待中）：等待各部完成

```

## 简报看板模板

```md
## 朝政进度

> 国之重器 ｜ <请示摘要> ｜ 模式：标准（先审后做）

- [x] 当前阶段：工部（代码执行）推进中
- [x] 执行权限：门下省已批准，工部可继续执行
- [x] 最近完成：门下省已返回批准，工部准予执行
- [ ] 当前阶段（进行中）：工部正在整理实现方案并核对影响面
- [ ] 当前阻塞（如无可省略）：等待礼部确认是否需要正式宣示
- [ ] 下一步：执行改动并进入奏折回呈
- [ ] 用户可选动作（如需要）：继续执行 / 仅出方案 / 跳过宣示
```

## 说明要求

每行说明应简短但具体。须提及以下之一：

- 当前执行者是谁，在做什么
- 正在等待什么前置条件
- 被什么阻塞
- 为什么跳过

好的示例：

- `尚书省正在审查代码逻辑与风险边界`
- `等待中书省返回奏折`
- `等待门下省批准`
- `已完成收件人解析与权限授予`
- `本案无需兵部介入`

避免模糊说明：

- `处理中`
- `分析中`
- `待定`

说明必须写清楚具体在做什么或缺少什么。

## 与团队蓝图的展示关系

朝政进度和团队蓝图在每次用户可见回复中按以下顺序展示：

1. 朝政进度（本文件定义的看板）
2. 团队蓝图（`team-blueprint-board.md` 定义的表格）
3. 一眼结论

两者的展示节奏同步：完整看板配完整蓝图，简报看板配简报蓝图。

## 语言规则

所有进度清单内容必须使用中文，不得使用英文。阶段名称、状态标记、说明文字全部使用中文。
